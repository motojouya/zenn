## 課題感
ドメイン駆動設計やMVCを盲目的に信じることで生じる歪みみたいなものはある気がしていて、それは明らかにしていきたい。  
それらを明らかにして、自分事として知識や技術が定着できればいいんだろうなという感じ。  
だが、そこに辿り着く前には、必ず違和感や痛みを伴う課題感があるはずだ。  

### ドメイン駆動設計におけるアプリケーション層とドメイン層
まず言及しておきたい課題感は、アプリケーション層とドメイン層という分け方についてだ。  
正直、これがよくわからない。ドメインの事情と、利用者の事情という説明をきいたことがあるが、利用をされるイメージがないとデータモデルは決まらない。  

例えばSNSにおいて、投稿内容のハンドリングはドメイン層に知識があってほしいと感じる。  
その投稿を見たか否か、つまり既読情報が保存されるデータモデルはどうだろうか。これは既読を管理したいというユーザ側の事情なのでは？などと考えてしまう。  
でも既読したか否かはデータベースに保存されるし、保存される前にドメインルールというフィルターを通るだろう。  
アプリケーション層と、ドメイン層というのは何が違うのか。  

### ドメインルールの表現力
また、ドメインルールというのも曖昧だ。  
データベースに依存しない、純粋なドメインモデルの実装クラスというのがあったとして、すべてのルールは表現できないように感じる。  
これは、Ruby on Railsの[記事](TODO)なんかでも言及されている。  

特定のオブジェクトというのは、そのデータモデルの中で表現できるルールは守ることができるが、他のオブジェクトとの関連の中にあるルールを表現することができない。  
具体的には、SNSの投稿は一意に特定することができる。というものだ。このためにはユニークキーをふるわけだが、ユニークか否かはデータベースに問い合わせて見なければわからない。  
UUIDは衝突耐性が非常に高いので安心して使えるが、衝突しないというルールはデータベースしか知らず、ドメインモデルの実装クラスだけでは実現できない。  

Web上に転がっている資料を読むと、このあたりはドメインルールではなく、データベースのルールという説明も見る。だが、それぞれの名前は関心事項ではない。  
問題は、そういったシステム上で開発時に把握しなければならない知識が、不用意に分散してしまうことが懸念だ。そういった制約全般はどのように参照して、どのように把握すればいいかわかりやすくあるべきだろう。  
理想的には、同じファイル上の、同じ名前空間で表現されるのが理想ではある。  

### MVC
MVCと言いつつ、SPAの登場によってUIを切り離して考えられるようになったので、ここはModel/Controllerという設計指針についての話になる。  
アプリケーションとドメインという分け方についてはすでに言及したので、Modelに書くか、Controllerに書くかという点については、同様の話題にだろう。  

問題はModelに機能が集中してしまうことだ。そしてModelへのメタ機能のためにフレームワーク側のコード量が膨大になり、実質的に知識が隠蔽されてしまう。  
OSSを使ってアプリケーションを作る時代に、隠蔽のレイヤというのは単なる可視性の問題ではない。そこに至るまでのコード量だろう。  

### MVCとForm
Ruby on Railsの[記事](TODO)においても、ModelとFormオブジェクトを分けろというのを見かけた。  
RailsはviewがあるフレームワークなのでFormといえばHTML Formだが、これは他の言語環境なら、単に入力値と捉えていい。入力値と保存するデータモデルを分けて、それぞれの役割を与えろということだ。  
コードが増えると、メタな機能を増やしたり、便利関数を増やす人もいるが、まず大事なのは、新しい役割のオブジェクトを作り、役割を整理してやることだろう。  
MVCは、そちらの方向に拡張できないわけではない。ただ人間は怠惰だし、MVCはそれらの方向性を示してくれない。なので便利関数を定義する方によってしまう。  

### MVCとValue Object
筆者は現職でLaravelを使っているが、入力値の定義がない配列を、100カラムを超えるModelに反映される処理のリファクタリングをしている。  
ここまで来ると本当に手がつけられない状態なわけだが、これらは先に言及したFormを使うことで緩和できる。  

また、MVCのModelはDBアクセス情報を含む非常に機能が膨大に詰め込まれたオブジェクトなので、DDDでいうところのValue Objectも切りづらい。  
100カラムのテーブルは数年単位の改修が必要そうだが、Value Objectの導入は本来はもっと短く適用できるはずではある。だがMVCのModelはそれもやりづらい。

### アプリケーション層とドメイン層に何を実装するのか
年末に、集約の単位でトップレベルのモジュールを分けて、その下にレイヤを切る。という[記事](TODO)を書いた。  
このレイヤの切り方こそが、アプリケーション層とドメイン層の区分けという課題感に対する回答となる。  

レイヤというか、特定の集約の配下には以下のディレクトリを切っている。  
- model  
  いわゆるドメインモデルでドメインルールが記述される。基本的にテーブルに対応する形を取り、後述するrecordの役割もこなす
- input  
  入力モデル。modelと違ってnullableになってたり入力を表現できる
- output  
  出力モデル。modelは他の集約の情報keyを持つが、実態は持てないので、別に実態を持てるモデルを用意する
- schema  
  DDL定義。集約配下に定義しているので、migration実行時にはTempディレクトリに移して実行する
- record  
  DBテーブル、SQLのSelect句に対応するオブジェクトだが、基本はmodelで事足りる。必要になるのは継続的な開発の中で、modelとtableが乖離してくるとき。
- store  
  いわゆるインフラ層で、sqlを定義する。  
- repository  
  storeの機能を使ってDBにしかない制約を含めて、ドメインルールに則り、ドメインの振る舞いを表現する。
- controller  
  複数のrepositoryを利用して、ドメインの振る舞いをラップして、実現したい機能を定義する。

ここではこれら全てに細かく言及はしない。それらはもっとテクニカルな後続記事で言及していく。  
ここで重要なのは、model,repository,controllerだ。  
MVCリスペクトなので、Model,Controllerという単語を利用し、Repositoryはバズワードと化しつつある気がするので適当に採用した。  
重要なのは、命名ではなく、役割をどう分けるかである。

### MVCとの共通点
repository+modelがmvcにおけるmodelになる。そして確かに、input,outputが不要なプロジェクトでさらなるわかりやすさを求めるならこうなる。
