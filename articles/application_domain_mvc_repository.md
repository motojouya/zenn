## 課題感
ドメイン駆動設計やMVCを盲目的に信じることで生じる歪みみたいなものはある気がしていて、それは明らかにしていきたい。  
それらを明らかにして、自分事として知識や技術が定着できればいいんだろうなという感じ。  
だが、そこに辿り着く前には、必ず違和感や痛みを伴う課題感があるはずだ。  

### ドメイン駆動設計におけるアプリケーション層とドメイン層
まず言及しておきたい課題感は、アプリケーション層とドメイン層という分け方についてだ。  
正直、これがよくわからない。ドメインの事情と、利用者の事情という説明をきいたことがあるが、利用をされるイメージがないとデータモデルは決まらない。  

例えばSNSにおいて、投稿内容のハンドリングはドメイン層に知識があってほしいと感じる。  
その投稿を見たか否か、つまり既読情報が保存されるデータモデルはどうだろうか。これは既読を管理したいというユーザ側の事情なのでは？などと考えてしまう。  
でも既読したか否かはデータベースに保存されるし、保存される前にドメインルールというフィルターを通るだろう。  
アプリケーション層と、ドメイン層というのは何が違うのか。  

### ドメインルールの表現力
また、ドメインルールというのも曖昧だ。  
データベースに依存しない、純粋なドメインモデルの実装クラスというのがあったとして、すべてのルールは表現できないように感じる。  
これは、Ruby on Railsの[記事](TODO)なんかでも言及されている。  

特定のオブジェクトというのは、そのデータモデルの中で表現できるルールは守ることができるが、他のオブジェクトとの関連の中にあるルールを表現することができない。  
具体的には、SNSの投稿は一意に特定することができる。というものだ。このためにはユニークキーをふるわけだが、ユニークか否かはデータベースに問い合わせて見なければわからない。  
UUIDは衝突耐性が非常に高いので安心して使えるが、衝突しないというルールはデータベースしか知らず、ドメインモデルの実装クラスだけでは実現できない。  

Web上に転がっている資料を読むと、このあたりはドメインルールではなく、データベースのルールという説明も見る。だが、それぞれの名前は関心事項ではない。  
問題は、そういったシステム上で開発時に把握しなければならない知識が、不用意に分散してしまうことが懸念だ。そういった制約全般はどのように参照して、どのように把握すればいいかわかりやすくあるべきだろう。  
理想的には、同じファイル上の、同じ名前空間で表現されるのが理想ではある。  

### MVC
MVCと言いつつ、SPAの登場によってUIを切り離して考えられるようになったので、ここはModel/Controllerという設計指針についての話になる。  
アプリケーションとドメインという分け方についてはすでに言及したので、Modelに書くか、Controllerに書くかという点については、同様の話題にだろう。  

問題はModelに機能が集中してしまうことだ。そしてModelへのメタ機能のためにフレームワーク側のコード量が膨大になり、実質的に知識が隠蔽されてしまう。  
OSSを使ってアプリケーションを作る時代に、隠蔽のレイヤというのは単なる可視性の問題ではない。そこに至るまでのコード量だろう。  

### MVCとForm
Ruby on Railsの[記事](TODO)においても、ModelとFormオブジェクトを分けろというのを見かけた。  
RailsはviewがあるフレームワークなのでFormといえばHTML Formだが、これは他の言語環境なら、単に入力値と捉えていい。入力値と保存するデータモデルを分けて、それぞれの役割を与えろということだ。  
コードが増えると、メタな機能を増やしたり、便利関数を増やす人もいるが、まず大事なのは、新しい役割のオブジェクトを作り、役割を整理してやることだろう。  
MVCは、そちらの方向に拡張できないわけではない。ただ人間は怠惰だし、MVCはそれらの方向性を示してくれない。なので便利関数を定義する方によってしまう。  

### MVCとValue Object
筆者は現職でLaravelを使っているが、入力値の定義がない配列を、100カラムを超えるModelに反映される処理のリファクタリングをしている。  
ここまで来ると本当に手がつけられない状態なわけだが、これらは先に言及したFormを使うことで緩和できる。  

また、MVCのModelはDBアクセス情報を含む非常に機能が膨大に詰め込まれたオブジェクトなので、DDDでいうところのValue Objectも切りづらい。  
100カラムのテーブルは数年単位の改修が必要そうだが、Value Objectの導入は本来はもっと短く適用できるはずではある。だがMVCのModelはそれもやりづらい。

### アプリケーション層とドメイン層に何を実装するのか
年末に、集約の単位でトップレベルのモジュールを分けて、その下にレイヤを切る。という[記事](TODO)を書いた。  
このレイヤの切り方こそが、アプリケーション層とドメイン層の区分けという課題感に対する回答となる。  

レイヤというか、特定の集約の配下には以下のディレクトリを切っている。  
- model  
  いわゆるドメインモデルでドメインルールが記述される。基本的にテーブルに対応する形を取り、後述するrecordの役割もこなす
- input  
  入力モデル。modelと違ってnullableになってたり入力を表現できる
- output  
  出力モデル。modelは他の集約の情報keyを持つが、実態は持てないので、別に実態を持てるモデルを用意する
- schema  
  DDL定義。集約配下に定義しているので、migration実行時にはTempディレクトリに移して実行する
- record  
  DBテーブル、SQLのSelect句に対応するオブジェクトだが、基本はmodelで事足りる。必要になるのは継続的な開発の中で、modelとtableが乖離してくるとき。
- store  
  いわゆるインフラ層で、sqlを定義する。  
- repository  
  storeの機能を使ってDBにしかない制約を含めて、ドメインルールに則り、ドメインの振る舞いを表現する。
- controller  
  複数のrepositoryを利用して、ドメインの振る舞いをラップして、実現したい機能を定義する。

ここではこれら全てに細かく言及はしない。それらはもっとテクニカルな後続記事で言及していく。  
ここで重要なのは、model,repository,controllerだ。  
MVCリスペクトなので、Model,Controllerという単語を利用し、Repositoryはバズワードと化しつつある気がするので適当に採用した。  
重要なのは、命名ではなく、役割をどう分けるかである。  

ポイントは、Repositoryだ。
Repositoryという単語は、設計の世界でバズワード化しているように感じる。なにかデータベースから取ってくるときのinterfaceになるというところで落ち着くことが多いように思う。  
でも、アプリケーション層、ドメイン層、インフラ層とした場合に、ドメイン層のinterfaceと言及されることが多いように思う。  

筆者は今回、明確にインフラ層のモジュールとしてstoreを定義した。Repositoryはドメイン層のものとして、ドメインロジックを表現するものとした。
そしてcontrollerはドメイン層を呼び出して、それらを関連付けるロジックを持つものとした。
controllerがrepository以外であるという定義なのであれば、Repositoryが表現するドメインとは何なのだろうか。
Repositoryは、特定の集約のロジックを示すものとした。集約と集約が関連する部分のロジックは、どちらかに実装するわけだが、それらを呼び出すのはcontrollerになる。

modelモジュールがあり、ここにはいわゆるドメインルール。ドメインロジックが実装される。
課題感で言及した、DBに問い合わせないとわからない制約については、ドメイン層のものであるRepositoryが担う。実際に問い合わせるsqlはstoreに実装されている。という形になる。

Repositoryという名前にこだわっているわけではない。多義的な言葉なので色々と連想してもらうためにあえて使っている。
インフラ層をラップして、ドメイン層が存在するわけだが、インフラ層つまりデータベースを含めて振る舞いをラップしていると捉えることもできる。
ドメインモデルという言葉もカジュアルに使ってきたが、これもバズワード的だ。ここで定義したmodelのことを指すことも多いが、より概念的なものを示し、modelを実装モデルと言及する記事もある。
ドメインモデルが概念モデルであるならば、それは実装モデルだけではなく、DB schemaにも反映されるだろう。実装モデルとデータベースの動きをラップして、利用可能なinterfaceを定義するrepositoryもまた、ドメインモデルの実装であると捉えることもできる。
なので、プロトタイププロジェクトでは、個人的には`Behavior`というモジュール名で呼んでいた。

いずれにしろ重要なのは名前ではなく、実装モデルとデータベースをラップして、特定の集約について概念的なドメインモデルを表現するというモジュールの切り方があるということだ。
そのモジュールの内部では、静的に表現されるドメインルールと、DBに問い合わせないとわからない制約を両方とじこめることができる。

これらの概念を実現しつつ、コードを書きやすいようにテンプレートプロジェクトは作った。
https://github.com/motojouya/ddd_go

ただ、プロトタイププロジェクトで作ったものをコピーして連れてきて、適当にfind&sedして編集したものなので、コンパイルは通らない。
筆者が利用する分には、適宜修正して使えるという代物になっている。

### Repositoryのinterface
Repositoryのinterfaceとしてよく言及されるのは、Repositoryは実装モデル、つまりここではmodelのオブジェクトを受け渡すというものだ。

実際に、`GetPostById`のような関数の返り値は、modelの構造体となる。
また、Postを集約ルートとして、Comment構造体のリストを持つという集約が表現されている場合、`GetPostById`の中でPostにCommentを紐づける。
```go
// この構造体がそのままテーブルだとしたとき、Commentsにテーブルのデータも取得して、この構造体に詰め込む
type Post struct {
  Title     string
  Content   string
  Comments []Comment
}
```

これは集約ルートを取得するRepositoryの基本的な実装だと思っている。

ただし、`UpdatePost`の引数は違っていて、入力値を受け付ける。ここではinputモジュールの構造体だ
```go
type PostContent struct {
  Content string
}
```

`UpdatePost`が記事内容のみを更新するという前提に立ち、ドメイン実装ではなく、入力値としてのinputを受け付ける。
入力値をどう反映させるのかは、ドメイン層の仕事だ。必然的にRepositoryにはロジックが生まれる。

`GetPostById`のような参照系のロジックでは、PostとCommentを紐づけて一つの構造体に詰め込むというロジック。  
`UpdatePost`のような更新系のロジックでは、ドメインの実装モデルを組み立てるロジックが発生する。
前者はORMを用いているとデータベースモジュールがやってくれるので便利であるが、今回のプロトタイプではORMを用いなかったので、そこは工夫して書いている。
そうすると必然的にインフラ層であるstoreは、本当に純粋に一つだけのsqlを投げる関数を提供するシンプルなモジュールになる。

これらの実装における細かいテクニックや考え方は、後続の記事で、参照系と更新系に分けて説明する。  
ここでは、概念として筆者がRepositoryと呼ぶモジュールは、ドメインモデルが表現され、それらのロジックが集まったものとして表現されるということ説明した。  

### MVCとの共通点
ここまでMVCには言及せず、ドメイン駆動設計っぽいが筆者の解釈によって表現されるモジュール設計について説明してきた。  
最初に言及したMVCへの伏線回収が必要だろう。  

今まで筆者は、今回説明したRepositoryのような概念を理解していなかった。  
Storeのようにsqlの発行はモジュールとして独立したものであってほしいと思っていたし、数あるDBアクセスを実現するライブラリは、それを便利にするために存在するというイメージでいた。  

だだ、Repositoryという概念でドメインモデルを表現しようとしたときに電気が走ったわけだ。「これActive Recordやん」と。
MVCのアーキテクチャではActive Recordと呼ばれるDBアクセスを実現するライブラリが利用される。  
だが、印象的な言葉でFatControllerという言葉がある。これはロジックがControllerに実装され、Modelに実装されていないのでドメインモデル貧血症になっているという状態を揶揄する言葉だ。
でもActive RecordがDBアクセスを実現するモジュールなのだとしたら、なぜドメインロジックを書かなければならないのか。これは強い違和感として持っていた。

でもなんのことはない。今回のRepositoryに対応する概念だとすれば、すっと理解できる。
今回のRepositoryは、DBアクセスを行うStore、静的なロジックを表現するmodelを利用して、DBを含めた概念的なドメインモデルの振る舞いを表現する。
Active Recordもそうなのである。だからActive Recordモデルには、ドメインロジックを表現しなくてはならない。

今回、Repository,Store,Modelと役割を分けたので、静的なドメインルールと、DBに問い合わせる制約はRepositoryからは明確だが、名前空間としては別れてしまう。
だが、Active Recordはこれらの制約を、同じ名前空間に表現できる。これに気づいたとき、本当に感動した。
これはRuby on Railsの[記事](TODO)でも言及されている。

ここまで共通点について言及し、MVCのActive Recordを上げまくってきたわけだが、筆者が今後選ぶことはないように思う。  
課題感でも言及したが、Active RecordはDB schemaとの紐づけが強く、Value Objectを切って行きづらいという性質を持っているからだ（できないわけではない）。  
DBのテーブルと1対1で紐づかないValue Objectというのは、訴求度の高いものではないが、仮にテーブルのカラムが100を超えてしまっているときなどは、処方箋としては効果てきめんである。  
そして、これは現在、筆者が仕事で向き合っているタスクでもある。  
また、Active Recordに機能が集中するため、どうしてもModelが複雑で長大になって、呼びづらくなる。そしてそれを解消するためのValue Objectはやりづらいというジレンマだ。  

また、input,outputというモジュールの存在を提示しておくのは、今後アプリケーションを育てていく際に、modelを過度に複雑にさせないために、実装を別に逃がしてやれるガイドラインにもなる。
素晴らしいからといって、自分に合うとは限らないのだ。
